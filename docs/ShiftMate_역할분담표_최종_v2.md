## 먼저 읽기: 백엔드와 프론트엔드가 뭔가요?

개발은 크게 두 영역으로 나뉩니다. 카페로 비유하면 이렇습니다.

```
손님 (사용자)
   ↕  화면을 보고 버튼을 누른다
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  프론트엔드 (홀)
  사용자가 직접 보는 화면
  버튼, 입력창, 목록, 캘린더
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
       ↕  "아메리카노 주문이요!" 라고 전달
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  백엔드 (주방)
  실제 데이터를 처리하는 곳
  저장, 계산, 검증, 응답
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
       ↕  데이터를 꺼내고 저장한다
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  데이터베이스 (창고)
  모든 정보가 저장된 곳
  직원 정보, 근무표, 요청 내역
```

### 둘이 어떻게 대화하나요? — API

프론트엔드와 백엔드는 **API(메뉴판)** 를 통해 대화합니다.

```
프론트엔드가 요청:  "GET /api/schedules?year=2025&month=5"
                   (2025년 5월 근무표 주세요)

백엔드가 응답:      { "schedules": [ ... ] }
                   (여기 있습니다)
```

한 사람이 **같은 기능의 백엔드(API)와 프론트엔드(화면)를 함께 담당**합니다.
자기가 만든 주방(API)에 자기가 홀(화면)을 연결하는 방식이에요.

## ShiftMate 전체 화면 지도

```
로그인 화면
비밀번호 변경 화면 (최초 로그인 시 강제 이동)
    │
    ├─── [관리자] ─────────────────────────────────────────────┐
    │         │                                               │
    │    관리자 대시보드                                        │
    │         │                                               │
    │    ┌────┴──────────────────────────────┐               │
    │    │              │                   │                │
    │  직원 관리        근무표 작성·조회      요청 관리          │
    │  화면 (모달)      화면 (공용 캘린더)    화면              │
    │    │                                   │                │
    │  근무 유형                             (휴무·휴가 승인)   │
    │  관리 화면 (모달)                       (교대 승인)       │
    │                                                        │
    └─── [직원] ─────────────────────────────────────────────┘
              │
         직원 대시보드
              │
         ┌────┴────────────────────────────────┐
         │                    │               │
       근무표 조회             휴무·휴가        교대 요청
       화면 (공용 캘린더)      신청+내역 화면    목록 화면
                                              교대 요청
                                              상세 화면
```

> 근무표 캘린더는 관리자·직원 공용 화면이에요.
> 관리자로 접속 시 전체 직원 근무표 + 확정 버튼 표시.
> 직원으로 접속 시 본인 근무표만 표시, 확정 버튼 없음.

## DB 테이블 전체 목록 (8개)

팀장이 Day 1 AM에 아래 8개 테이블을 모두 생성합니다.
멤버들은 자기 담당 테이블 구조를 이 목록에서 확인하세요.

| 테이블 | 담당 | 설명 |
|---|---|---|
| `users` | 팀장 | 직원·관리자 계정 |
| `shift_types` | 멤버 1 | 근무 코드 (D/E/N/OFF/VAC 등) |
| `schedules` | 멤버 2 | 직원별 날짜별 근무 배정 |
| `schedule_periods` | 멤버 2 | 월 단위 확정 상태 |
| `swap_requests` | 팀장 | 교대 요청 |
| `swap_proposals` | 팀장 | 교대 제안 (swap_requests 1:N) |
| `time_off_requests` | 멤버 3 | 휴무·휴가 신청 |
| `system_settings` | 팀장 | 검증 수치·교대 설정 (단일 행) |

> `system_settings`는 관리자 설정 화면 없이 초기값으로 자동 생성됩니다.
> 애플리케이션 기동 시 행이 없으면 기본값으로 INSERT합니다.
> 멤버 2의 검증 API와 팀장의 교대 연차 검증이 이 테이블을 참조합니다.

#### 담당 화면

| 화면 | 설명 |
|---|---|
| 🔐 로그인 화면 | 사번 + 비밀번호 입력, 로그인 처리 |
| 🔑 비밀번호 변경 화면 | 최초 로그인 시 강제 이동, 변경 처리 |
| 🔄 교대 요청 목록 화면 (직원) | 내가 보낸 요청 + 받은 요청 목록 |
| 🔄 교대 요청 상세 화면 (직원) | 요청 상세 확인, 상태에 따라 버튼 다르게 표시 |

#### 교대 요청 상세 화면 — 상태별 버튼 동작

```
[직원 A — 요청자 입장, pending 상태]
┌────────────────────────────────────┐
│ 교대 요청 상세                      │
│ 5월 15일 D근무 → 교대 원함          │
│ 상태: 대기 중                       │
│ 들어온 제안 없음                     │
└────────────────────────────────────┘

[직원 A — 요청자 입장, accepted 상태]
┌────────────────────────────────────┐
│ 교대 요청 상세                      │
│ 5월 15일 D근무 → 교대 원함          │
│ 상태: 제안 선택 대기                 │
│ 이영희 — 5/20 N근무   [선택]        │
│ 박민준 — 5/22 D근무   [선택]        │
└────────────────────────────────────┘

[직원 B — 제안자 입장, pending 상태]
┌────────────────────────────────────┐
│ 교대 요청 상세                      │
│ 김철수 — 5월 15일 D근무 교대 원함   │
│ 상태: 대기 중                       │
│              [제안하기]             │
└────────────────────────────────────┘
```

#### 담당 흐름 — 교대 요청 전체 흐름

```
[직원 A] 교대 요청 목록 화면 → "새 요청" 버튼
         → 내 시프트 선택
                  ↓
         백엔드: 요청 저장, 연차 범위 계산, 시프트 잠금
         (연차 범위 = system_settings.swap_years_range 기반)
                  ↓
[직원 B] 교대 요청 목록 화면에서 요청 확인
         → 상세 화면 → "제안하기" 클릭
                  ↓
         백엔드: 연차 범위 검증, 제안 저장, 시프트 잠금
                  ↓
[직원 A] 상세 화면에서 제안 목록 확인 → "선택" 클릭
                  ↓
         백엔드: 나머지 제안 자동 거절
                  ↓
[관리자] 요청 관리 화면에서 교대 행 확인
         → "승인" 또는 "반려" 클릭
                  ↓
         백엔드: 두 근무표 교환 (단일 트랜잭션), 잠금 해제
```

#### 담당 백엔드 (API)

| 구분 | 하는 일 | 비고 |
|---|---|---|
| DB 전체 세팅 | 8개 테이블 생성, 제약 조건, 초기 관리자 계정 | system_settings 초기값 포함 |
| 로그인 API | 사번+비밀번호 확인, 토큰 발급 | |
| 내 정보 조회 API | 로그인된 사용자 정보 반환 | |
| 로그아웃 API | 토큰 무효화 | |
| 비밀번호 변경 API | 현재 비밀번호 확인 후 변경 | |
| 교대 요청 생성 API | 중복 체크, 날짜 검증, 시프트 잠금 | |
| 교대 요청 목록 API | swap_request_id, requester_id, requester_schedule_id, status 반환 | |
| 교대 제안 생성 API | 연차 범위 검증 (system_settings 참조), 제안자 시프트 잠금 | |
| 교대 제안 선택 API | 나머지 제안 자동 거절 | |
| 교대 승인 API | 두 근무표 교환 (단일 트랜잭션) | |
| 교대 반려 API | 두 시프트 잠금 해제 | |

> Swap API는 schedule_id만 반환해요.
> 날짜·근무코드·색상은 FE에서 근무표 조회 API를 별도 호출하여 매핑합니다.
> ```
> 1. Swap 목록 API 호출 → requester_schedule_id, proposer_schedule_id 받음
> 2. 근무표 조회 API 호출 → schedule_id로 날짜·shift_code 조회
> 3. 근무유형 목록 API 호출 → shift_code로 색상 조회
> 4. FE에서 매핑하여 "김철수 — 5/15 D근무" 형태로 표시
> ```
> ⚠ 팀장 FE에서 멤버 2 BE(근무표 조회)와 멤버 1 BE(근무유형 목록)에 의존해요.

#### 근무표 작성 화면 구조

```
┌──────────┬────┬────┬────┬─────┬────┐
│ 직원\날짜 │ 1일│ 2일│ 3일│ ... │말일│
├──────────┼────┼────┼────┼─────┼────┤
│ 김철수   │[D▼]│[N▼]│[OFF]│ ...│[D▼]│
│ 이영희   │[E▼]│[D▼]│[N▼]│ ...│[OFF]│
│ 박민준   │[OFF]│[E▼]│[D▼]│...│[N▼]│
└──────────┴────┴────┴────┴─────┴────┘
                              [저장] [이번 달 확정]

- 행: 직원 목록 (멤버 1 BE의 직원 목록 API에서 가져옴)
- 열: 해당 월 날짜 (1일 ~ 말일)
- 각 셀: 근무 코드 선택 드롭다운
         (멤버 1 BE의 근무유형 목록 API에서 가져옴)
- 미배정 셀: 빈칸 표시
- 저장: 변경된 전체 셀을 한 번에 저장
        (멤버 2 BE의 근무표 일괄 저장 API 호출)
- 이번 달 확정: 멤버 2 BE의 월 확정 API 호출
```

### 🔵 멤버 2 — "근무표 저장/조회/확정 + 캘린더 화면 + 직원 대시보드"

> ShiftMate의 핵심인 근무표 데이터 처리를 담당합니다.
> 캘린더 화면은 관리자·직원 공용으로 사용하며
> 권한에 따라 다르게 동작해요.

#### 담당 화면

| 화면 | 설명 |
|---|---|
| 📅 근무표 캘린더 조회 화면 (공용) | 관리자: 전체 직원 + 확정 버튼. 직원: 본인만 표시 |
| 🏠 직원 대시보드 | 오늘 내 근무, 이번 주(월~일) 근무 일정 표시 |

#### 캘린더 화면 권한 분기

```
관리자로 접속 시:
┌─────────────────────────────────────────┐
│  ◀  2025년 5월  ▶      [필터: 전체 ▼]   │
│  ⚠ 이 달은 아직 미확정입니다. [이번 달 확정]│
│  [캘린더 — 전체 직원 근무표]             │
└─────────────────────────────────────────┘

직원으로 접속 시:
┌─────────────────────────────────────────┐
│  ◀  2025년 5월  ▶                       │
│  [캘린더 — 본인 근무표만]                │
│  (확정 버튼 없음, 필터 없음)             │
└─────────────────────────────────────────┘
```

#### 캘린더 색상 표시 방식

```
캘린더 화면 진입 시 두 API를 순서대로 호출:
1. 멤버 1 BE의 근무유형 목록 API → 코드별 색상 정보 받음
   예: { "D": "#3B82F6", "N": "#EF4444", "OFF": "#9CA3AF" }

2. 멤버 2 자신의 근무표 조회 API → 날짜별 shift_code 받음
   예: [{ "date": "2025-05-01", "shift_code": "D" }, ...]

3. FE에서 매핑하여 색상 적용
   예: shift_code "D" → color "#3B82F6" → 캘린더에 파랑으로 표시

⚠ 멤버 1 BE의 근무유형 목록 API가 완성되어야
  색상 표시가 가능합니다.
  완성 전: Mock 색상 데이터로 먼저 구현
  완성 후: 실제 API 호출로 교체
```

#### 담당 흐름

```
[멤버 1 화면] 근무표 작성 화면에서 "저장" 클릭
                  ↓
         멤버 2 백엔드: 근무표 일괄 저장
         (해당 날짜 레코드 있으면 수정 / 없으면 새로 추가)
                  ↓
[관리자] 작성 완료 후 "이번 달 확정" 버튼 클릭
         (근무표 작성 화면 또는 캘린더 조회 화면에서 가능)
                  ↓
         멤버 2 백엔드: 검증 실행 → 결과 반환
         → 경고 없으면 확정 / 경고 있으면 모달 표시
         (확정 시 schedule_periods 테이블에 레코드 생성)
                  ↓
[직원] 캘린더 화면 → 이번 달 색상별 근무 코드 확인

### 🟡 멤버 3 — "휴무·휴가 신청 + 요청 관리 + 관리자 대시보드"

> 직원이 휴무를 신청하고 관리자가 처리하는 흐름 전체를 담당합니다.
> 관리자 요청 관리 화면에서 교대 요청 승인도 함께 표시됩니다.

#### 담당 화면

| 화면 | 설명 |
|---|---|
| 📋 휴무·휴가 신청 + 내역 화면 (직원) | 상단: 신청 폼. 하단: 내 요청 내역 목록 |
| 📋 요청 관리 화면 (관리자) | 휴무·휴가 + 교대 요청 통합 목록, 승인·반려 처리 |
| 📊 관리자 대시보드 | 전체 직원 수, 오늘 근무자, 휴가자, 승인 대기 건수, 이번 주 근무 현황 |

#### 담당 흐름 1 — 휴무·휴가 신청 + 내역 (통합 화면)

```
[직원] 휴무·휴가 신청 + 내역 화면
┌─────────────────────────────────────┐
│ 신청하기                             │
│ 유형: [휴무 OFF ▼]                  │
│ 날짜: [2025-05-10] ~ [2025-05-10]  │
│ 사유: ________________              │
│                          [신청]     │
├─────────────────────────────────────┤
│ 내 요청 내역                         │
│ 5/10 OFF  대기 중    [취소]          │
│ 4/22 VAC  승인됨                    │
│ 4/05 OFF  반려됨                    │
└─────────────────────────────────────┘

신청 완료 후: 폼 초기화 + 내역 목록 새로고침
```

#### time_off_requests 테이블 구조 (멤버 3 담당)

```
날짜는 단일 컬럼이 아닌 start_date + end_date 두 컬럼으로 저장해요.
단일 휴무(하루)인 경우 start_date와 end_date 값이 동일합니다.

예:
- 5/10 하루 OFF → start_date: 2025-05-10, end_date: 2025-05-10
- 5/20~22 VAC   → start_date: 2025-05-20, end_date: 2025-05-22

type: ENUM('OFF', 'VAC') — 이 두 가지만 허용
status: ENUM('pending', 'approved', 'rejected', 'canceled')
```

#### 휴무 신청 비즈니스 규칙

```
✅ 중복 신청 차단
   같은 날짜 범위에 이미 pending 상태 신청이 있으면 새 신청 거부
   → HTTP 400 + {"detail": "해당 날짜에 이미 대기 중인 신청이 있습니다"}

✅ 취소 가능 조건
   pending 상태일 때만 취소 가능
   approved / rejected 상태에서는 취소 버튼 미표시
   → API 레벨에서도 pending이 아니면 HTTP 400 반환
```

#### 담당 흐름 2 — 관리자 요청 관리 화면 (통합)

```
┌─────────────────────────────────────────────┐
│ 요청 관리                        [전체 ▼]    │
├──────────┬────────────┬──────────┬──────────┤
│ 유형     │ 신청자     │ 내용     │ 처리     │
├──────────┼────────────┼──────────┼──────────┤
│ 휴무     │ 김철수     │ 5/10 OFF │[승인][반려]│ ← 멤버 3 API
│ 교대 요청│ 이영희     │ D↔N 5/15 │[승인][반려]│ ← 팀장 API
│ 휴가     │ 박민준     │ 5/20~22  │[승인][반려]│ ← 멤버 3 API
└──────────┴────────────┴──────────┴──────────┘
```

#### 담당 백엔드 (API)

| 구분 | 하는 일 | 비고 |
|---|---|---|
| 휴무·휴가 신청 API | 유형·start_date·end_date·사유 저장 | 중복 신청 차단 |
| 내 요청 목록 API | 내가 신청한 것들 반환 | |
| 전체 요청 목록 API | 관리자용 전체 목록 반환 | |
| 요청 승인 API | 상태를 "승인"으로 변경 | |
| 요청 반려 API | 상태를 "반려"로 변경 | |
| 요청 취소 API | pending 상태인 경우에만 취소 가능 | |
| 관리자 대시보드 API | 전체 현황 반환 | 두 테이블 합산 |

#### 관리자 대시보드 API 반환 항목

```
{
  "total_employees": 전체 직원 수,
  "today_working": 오늘 is_work_day=true 코드 배정 직원 수,
  "today_on_leave": 오늘 VAC 코드 배정 직원 수,
  "pending_requests": pending 대기 건수 합산
    (= time_off_requests.pending + swap_requests.pending + swap_requests.accepted),
  "this_week_schedule": 이번 주(월~일) 날짜별 근무자 수 배열
}
```

#### 타임라인

| 일차 | 작업 |
|---|---|
| Day 1 PM | 휴무·휴가 API × 6, 관리자 대시보드 API |
| Day 2 AM | 관리자 대시보드 화면 |
| Day 2 PM | 휴무·휴가 신청 + 내역 통합 화면 |

  swap_requests 테이블 구조는 이 문서 상단 "DB 테이블 전체 목록" 확인

## Claude Code PRO 활용 가이드

### 프로젝트 디렉터리 구조 (확정)

#### 백엔드

```
backend/
  app/
    api/            # 라우터 파일 (employees.py, schedules.py 등)
    models/         # SQLAlchemy 모델 (user.py, schedule.py 등)
    schemas/        # Pydantic 요청·응답 스키마
    dependencies/   # auth.py (require_admin, get_current_user)
    core/           # config.py (환경변수), security.py (JWT)
  alembic/          # DB 마이그레이션
  main.py
  requirements.txt
```

#### 프론트엔드

```
frontend/src/
  api/
    client.js       # Axios 인스턴스 (모든 API 호출은 여기서)
    mocks/          # Mock 데이터 파일
  pages/
    admin/          # 관리자 전용 화면
    employee/       # 직원 전용 화면
  components/       # 공용 컴포넌트 (Button, Modal, Table 등)
  context/
    AuthContext.jsx # 인증 상태 관리
```

### 인증 방식 (확정)

```
- 토큰: JWT
- 저장: localStorage
- 헤더: Authorization: Bearer <token>
- 상태관리: Context API

// context/AuthContext.jsx
const { user, token, login, logout } = useAuth()

// user 구조
{
  user_id,
  name,
  role,                  // 'admin' | 'employee' — Route Guard용
  is_initial_password,   // true면 /change-password로 강제 이동
  employee_no,
  years_of_experience
}

// 사용 예시
const { user } = useAuth()
if (user.role === 'admin') { /* 관리자 전용 */ }
if (user.is_initial_password) { navigate('/change-password') }
```

```
내가 만든 직원 등록 API야.
아래 항목을 기준으로 문제를 찾아서 파일에 바로 반영해줘.

```

### Mock에서 실제 API로 바꿀 때

```
지금 Mock 데이터로 화면을 만들었어.
아래 API가 완성됐으니 실제 API 호출 코드로 바꿔줘.

반드시 src/api/client.js 의 apiClient를 사용해줘.
axios.post() 직접 호출은 쓰지 마.

API 명세:
- URL: POST /api/admin/schedules/bulk
- 요청: [{ user_id, work_date, shift_type_id }]
- 성공 응답: { success: true }
- 실패 응답: { detail: "에러 메시지" }
- 실패 시 화면에 에러 토스트 메시지 표시

로딩 중에는 버튼 비활성화 처리도 추가해줘.

파일 위치: frontend/src/pages/admin/SchedulesPage.jsx
현재 Mock 코드 위치: src/api/mocks/schedules.js
```

## 완성 기준 체크리스트

### 팀장 Day 1 완료 체크리스트 (멤버 착수 조건)

```
□ DB 마이그레이션 실행 가능 (8개 테이블 + system_settings 초기값)
□ POST /api/auth/login 동작 확인
□ FE 프로젝트 npm run dev 실행 확인
□ 로그인 화면 → 관리자/직원 분기 이동 동작
□ src/api/mocks/ Mock 파일 전체 FE 레포 커밋
□ API 명세서 공유 (슬랙 or 노션)
```

### 데모 기준 최종 체크리스트

```
□ 로그인 → 관리자 대시보드 진입
□ 대기 요청 건수 표시 (휴무 + 교대 합산)
□ 직원 등록 → 임시 비밀번호 모달 표시 → 복사
□ 근무 유형 추가 (코드, 색상 포함)
□ 근무표 작성 → 저장
□ 근무표 검증 경고 표시 (연속 야간 등)
□ 근무표 월 확정 (작성 화면 또는 캘린더 화면에서)
□ 근무표 캘린더 조회 (색상 코드 적용)
□ 관리자 캘린더에서만 확정 버튼 표시 확인
□ 직원 계정으로 로그인 → 직원 대시보드 → 내 근무표 조회
□ 직원 캘린더에서 확정 버튼 없음 확인
□ 직원이 교대 요청 생성
□ 다른 직원이 교대 제안
□ 요청자가 제안 선택
□ 관리자가 교대 승인 → 두 직원 근무표 교환 확인
□ 직원이 휴무 신청 → 내역 즉시 표시
□ 관리자가 휴무 승인 처리
```
